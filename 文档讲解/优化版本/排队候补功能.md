
**文档概述**
- 在现有 Redis/Lua 秒杀预扣、Kafka 异步落单、`VoucherOrderServiceImpl#createVoucherOrderV2` 的基础上，新增“候补/排队”能力：当库存为 0 时用户可加入候补队列；一旦出现库存，系统按队列顺序自动发券（创建订单），无需用户确认和支付。
- 前端进入店铺详情页时仅进行一次性状态查询（已购 + 候补 + 库存），不进行轮询；用户点击“候补/排队”时调用后端接口。

**业务目标**
- 库存为 0、未购买的用户可以候补。
- 释放库存（取消订单/补货）或当前就有库存时，按队列顺序自动发券，直接创建订单。
- 保证公平先来先得、一人一单、操作幂等，保持与现有架构一致。

**前端交互（ShopDetail.vue）**
- 初始化一次性查询每张券的状态并渲染：
  - 已购：按钮置灰为“已领取”。
  - 库存 > 0：显示“限时抢购”（保持现状）。
  - 库存 <= 0 且未购：
    - 未在候补：显示“候补/排队”按钮。
    - 已在候补：置灰为“候补中”，可提供“取消候补”按钮。
  - 候补命中后：后端已发券，返回 `orderId` 或 `isPurchased=true`，前端即显示“已领取”。
- 前端状态映射：
  - `purchasedMap: { [voucherId]: boolean }`
  - `waitlistStatusMap: { [voucherId]: 'none' | 'waiting' | 'success' | 'canceled' }`
- 初始化流程建议：
  - 调用批量接口一次性获取：`isPurchased + waitlistStatus + stock`，避免 N+1。
  - 点击“候补/排队”时仅调用一次 `join`。

**接口总览**
- `POST /voucher/waitlist/join`：加入候补；若当前有库存则直接发券并返回订单号。
- `GET /voucher/waitlist/status`：查询当前用户对单张券的候补状态。
- `POST /voucher/waitlist/status/batch`：批量查询多张券的候补状态（推荐用于页面初始化）。
- `POST /voucher/waitlist/cancel`：取消候补（仅候补中可取消）。
- 可选合并查询：`POST /voucher/waitlist/overview`：批量返回“已购 + 候补 + 库存”，前端一次性渲染。

**状态枚举**
- `WAITING`：已在候补队列。
- `SUCCESS`：候补命中，系统已发券（创建订单）。
- `CANCELED`：用户取消候补。
- 未参与候补：返回时 `status` 为空或不返回。

**返回封装**
- 统一使用 `Result<T>`：
  - `success: boolean`
  - `data: T`
  - `errorMsg: string`（失败时）
  - 可选 `code: string | number`（与 `BaseCode` 对齐）

**接口规格与示例**

- `POST /voucher/waitlist/join`
  - 入参（JSON）
    - `voucherId: Long` 必填
    - `source: String` 可选（埋点，如 `shop_detail`）
  - 鉴权
    - 从登录态获取 `userId`（`UserHolder`）
  - 出参（`Result<{ status: "SUCCESS" | "WAITING", orderId?: Long }>`）
    - `SUCCESS`：立即发券（Redis `DECR`），返回 `orderId`
    - `WAITING`：入队成功
  - 失败示例
    - `success=false, errorMsg="活动未开始/已结束"`
    - `success=false, errorMsg="当前会员级别不满足参与条件"`
    - `success=false, errorMsg="您已购买该券，不能重复候补"`
    - `success=false, errorMsg="参数非法/优惠券不存在"`

- `GET /voucher/waitlist/status?voucherId={id}`
  - 入参（Query）
    - `voucherId: Long` 必填
  - 鉴权
    - 后端获取 `userId`
  - 出参（`Result<{ status?: "WAITING" | "SUCCESS" | "CANCELED", orderId?: Long, isPurchased: boolean }>`）
    - 未参与候补：`status` 可为空或不返回
    - `SUCCESS/已购` 时可返回 `orderId`

- `POST /voucher/waitlist/status/batch`
  - 入参（JSON）
    - `voucherIds: List<Long>` 必填（建议 ≤ 50）
  - 鉴权
    - 后端获取 `userId`
  - 出参（`Result<{ entries: Array<{ voucherId: Long, status?: "WAITING" | "SUCCESS" | "CANCELED", orderId?: Long, isPurchased: boolean, stock?: Integer }> }>`）

- `POST /voucher/waitlist/cancel`
  - 入参（JSON）
    - `voucherId: Long` 必填
  - 鉴权
    - 后端获取 `userId`
  - 出参（`Result<Boolean>`）
    - 仅 `WAITING` 可取消；`SUCCESS/已购` 返回失败并提示“已发券/已领取，无法取消”
    - 取消不变更库存

- 可选：`POST /voucher/waitlist/overview`
  - 入参（JSON）
    - `voucherIds: List<Long>` 或 `shopId: Long`
  - 出参（`Result<{ entries: Array<{ voucherId, status?, orderId?, isPurchased, stock }> }>`）

**Redis/Lua 设计**
- 键约定（每个 `voucherId` 一套）：
  - `WAITLIST:Z:{voucherId}`：ZSET，成员 `userId`，score 入队时间（先来先得）
  - `WAITLIST:USERS:{voucherId}`：SET 去重（防重复入队）
  - `WAITLIST:STATUS:{voucherId}`：HASH，`userId -> WAITING | SUCCESS | CANCELED`
- 复用现有键：
  - `SECKILL_STOCK_TAG_KEY`（库存标记）
  - `SECKILL_USER_TAG_KEY`（用户参与标记，一人一单）
  - `SECKILL_TRACE_LOG_TAG_KEY`（日志/埋点）
- Lua 操作（建议新增在 `SeckillVoucherOperate`）：
  - 加入候补（join）
    - 时间窗/等级校验（等级建议业务层先做）
    - 已购/已在队列校验
    - `stock > 0`：`DECR stock`、写 `SECKILL_USER_TAG_KEY`，生成 `orderId`、返回 `SUCCESS + orderId`
    - `stock <= 0`：`SADD WAITLIST:USERS`、`ZADD WAITLIST:Z`、`HSET WAITLIST:STATUS WAITING`，返回 `WAITING`
  - 候补匹配（库存释放/补货事件触发）
    - 循环：每次 `DECR stock` → `ZPOPMIN` 队首 → 写 `SECKILL_USER_TAG_KEY` → 输出 `userId+orderId+traceId` 给业务层，随后发 Kafka
    - 队列为空则 `INCR stock` 回滚
  - 取消候补（cancel）
    - `SREM WAITLIST:USERS`、`ZREM WAITLIST:Z`、`HSET WAITLIST:STATUS CANCELED`（不变更库存）

**Kafka 与订单落地**
- 候补命中后（`join` 立即命中或匹配脚本命中）：
  - 发送 `SeckillVoucherMessage`（复用或扩展现有格式）：`userId, voucherId, orderId, traceId, beforeQty, deductQty, afterQty`
  - 消费端走 `VoucherOrderServiceImpl#createVoucherOrderV2`：
    - DB 扣减库存并创建订单、写 `VoucherOrderRouter`、短期写入 Redis（`DB_SECKILL_ORDER_KEY`）供前端显示
- 失败兜底：
  - DB 扣减失败 → 消息侧回滚 Redis，并继续匹配下一位候补（确保一致性）

**幂等与一致性**
- 幂等：
  - `SECKILL_USER_TAG_KEY` 保证一人一单，避免重复发券
  - 候补入队用 `WAITLIST:USERS:{voucherId}` 去重
  - `RepeatExecuteLimit` 可用于候补接口与消息消费者，防止重复处理
- 一致性：
  - Redis 扣减与队列匹配在 Lua 中原子执行
  - 落单失败由消息侧回滚 Redis，保持最终一致

**安全与校验**
- 活动时间窗：未开始/已结束直接拒绝
- 用户等级：`verifyUserLevel(seckillVoucher, userId)`（沿用现有逻辑）
- 参数合法性：券存在与否，ID 类型校验
- 已购校验：复用 `getVoucherOrderIdByVoucherId` 或通过 `SECKILL_USER_TAG_KEY`

**数据与监控**
- 指标：
  - 候补队列长度、自动发券成功率、DB 扣减失败率
  - 库存释放来源：取消/补货
- 报警：
  - 长时间队列不消化、消息消费异常、DB 扣减失败率异常
